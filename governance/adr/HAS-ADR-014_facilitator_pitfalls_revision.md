# Title: HAS-ADR-014: Facilitator Principlesを「壊れやすいポイントの地図」へ転換し、Facilitator Pitfallsとしてリネーム

## Status
Accepted

## Context

`04_facilitator_principles.md` (v1.0.0) は、HAS初期に作成された文書であり、
思想を守るために「強く」「断定的に」書かれていた。

しかしHAS v2.0 Finalの完成により、以下の問題が顕在化した：

1. **Must構造の再生産**
   - 「必須条件」「使用してはならない」という強制的記述
   - HASが批判するMust/Fear構造を、自ら体現していた

2. **Level 0との論理矛盾**
   - ADR-003で「何もしないことは正規の実践」と定義
   - しかしv1.0.0では「守れない限り使用してはならない」と記述
   - これは構造的に矛盾する

3. **Failure Mode Eの誘発**
   - 完璧主義ループ（自己監視の宗教化）を誘発する記述
   - 「整っていない限り介入しない」が強迫観念に転化しやすい

4. **科学的主張の不整合**
   - 「神経系レベルで起きる物理現象」という記述
   - 実証的根拠が不明確であり、決定論的に過ぎる

5. **Being as Interventionの概念矛盾**
   - HAS Kernelは「非介入」を憲法第一条とする
   - しかしタイトルで「介入」を肯定的に使用していた

6. **ファイル名と内容の乖離**
   - 「Principles（原則）」という名称が、規範・正解集という誤読を招く
   - 実際の内容は「壊れ方の地図」であり、名称がミスリーディング

## Problem

HASが成熟した現在、初期防衛層として機能していたこの文書は、
**HAS自身の思想と衝突するリスク**を孕んでいる。

特に「Principles」という名称は：
- 権威性を呼ぶ
- 正解集だと誤解される
- Must構造を勝手に補完される

文書の存在意義を再定義せずに放置すると：
- 実践者が萎縮し、Failure Mode Eに陥る
- HASが「難しすぎて使えない」という評判を招く
- 思想的整合性が失われる

## Decision

`04_facilitator_principles.md` を大幅に縮小し、
**「壊れやすいポイントの地図」へ転換する。**

同時に、ファイル名を **`04_facilitator_pitfalls.md`** にリネームする。

具体的には：

1. **断定・命令の削除**
   - 「必須条件」→ 削除
   - 「使用してはならない」→ 「Level 0に戻ることが適切」
   - 「必ず」「崩壊する」「唯一」→ 削除

2. **Failure Modesとの統合**
   - Facilitator特有の壊れ方を、Failure Modesへ参照
   - 「こうすべき」ではなく「こうなったら戻る」の記述

3. **科学的主張の削除**
   - 「神経系レベルの物理現象」→ 削除
   - 「影響を及ぼす」という慎重な表現に変更

4. **「介入」という語の最小化**
   - HAS Kernelの「非介入」は憲法語彙として強すぎる
   - 運用文書では「影響」「関与」「Presence」を優先
   - 再定義を試みず、使用そのものを避ける

5. **Level 0の正当化**
   - 何もしないことは失敗ではないと明示
   - 完璧主義ループを構造的に防ぐ

6. **沈黙の美化への予防**
   - P03の原則を維持しつつ、緊急停止の優先を明記
   - 「沈黙こそ正義」という誤読を防ぐ

7. **ファイル名の変更**
   - `principles` → `pitfalls`
   - 名称と内容の乖離を解消
   - 規範臭を完全に除去

## Rationale

HASの成熟には、2つのフェーズがある：

- **初期（v1.0）**: 思想を守るために「強く」書く
- **成熟（v2.0）**: 思想を守るために「削る」

現在は後者のフェーズに入っている。

強く書くことで、かえってHASの非Must思想が損なわれる段階に来た。

また、Failure Modesという「否定形で書ける場所」が確立された今、
規範を前面に押し出す必要はなくなった。

特に重要なのは：
- 「介入」という語の再定義を試みないこと
- HAS Kernelの憲法語彙に、運用文書が触れないこと
- **名称と内容を一致させること**

HASは言語設計がコアである。
名称がズレた瞬間に、思想が歪む。

「Facilitator Principles」という名称は：
- 権威性を呼ぶ
- 正解集だと誤解される
- Must構造を勝手に補完される

今の内容と真逆である。

「Pitfalls（落とし穴）」という名称は：
- 規範臭がゼロ
- Failure Modesと自然に並ぶ
- 「狂ったら戻る座標」という実体を正確に表す

人間は放っておくとすぐ宗教を作る。
語彙の生存を最小化することが、誤読の予防になる。

## Pros & Cons

**Pros**
- \+ Must構造の消滅
- \+ Level 0との論理的整合
- \+ Failure Mode Eの予防
- \+ 実践者の心理的負担軽減
- \+ HAS思想との完全な整合
- \+ 「介入」という語彙の誤読リスク低減
- \+ 沈黙の美化→ネグレクトへの予防
- \+ 名称と内容の一致
- \+ 規範臭の完全除去

**Cons**
- − 既存読者が「あの文書はどこに？」と混乱する可能性
- − ファイル名変更により、外部リンクが切れる可能性

**対策**
- v1.0.0をアーカイブとして残す
- README / Architecture Mapで位置づけの変更を明示
- このADRで判断理由を明確に記録
- リダイレクトや移行メモを適宜配置

## Alternatives Considered

### B案：ファイル名は維持、内容のみ変更
- 理由：既存リンクが保持される
- 却下：名称と内容の乖離が残り、誤読リスクが継続する

### C案：`04_facilitator_risk_map.md`
- 理由：Architecture Mapと語彙が揃う
- 却下：抽象的すぎる。「Pitfalls」の方が生々しく、実体に近い

### D案：完全削除 + 各文書への分散配置
- 理由：思想的には最もクリーン
- 却下：将来の最終整理として温存（今はやらない）

## Consequences

この改訂により：

1. **Facilitator Pitfallsは「制約条件」ではなく「運用リスクマップ」に**
   - 必読ではなく、壊れやすいポイントを知るためのガイド
   - Architecture Map Layer Bの位置づけを「安全仕様＋運用事故カタログ」に変更

2. **Failure Modesが規範の受け皿に**
   - 否定形で書けるため、HASの非Must思想と整合する

3. **HASの非階層性が明確に**
   - Facilitator専用文書という響きが薄まる
   - 個人利用可能性（FAQ Q13）との整合性が高まる

4. **防衛姿勢の解除**
   - HASはもう、強く縛らないと壊れる段階を過ぎている
   - 削れる体系は、成熟している証拠

5. **語彙の生存を最小化**
   - 「介入」という語を運用文書から排除
   - 憲法語彙への接触を減らし、誤読の導線を断つ

6. **名称の正直さ**
   - 中身と名前がズレない
   - 思想が言語設計に正確に反映される
