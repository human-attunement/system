# ADR-024 HASにおける「壊れる」語彙の廃止と「選択可能性」中心への再定義

## Status
Accepted

## Context
HAS 2.0 公開後の文書精査により、「壊れる／壊さない／壊れやすい」といった語彙が、
読者・実践者に対して診断・修復・介入・メソッド化を誘発するリスクが明確になった。

HASは本来、
- 失敗を防ぐ体系でも
- 関係を修復する体系でも
- 正しい行為を指示する体系でもない

にもかかわらず、「壊れる」という語が、
設計思想と矛盾する解釈を生みやすいことが判明した。

## Problem

読者は「壊れる」という表現に反応するのではないか？

- 「何が壊れるの？」
- 「壊れたら、直さなければいけない」
- 「壊れて何が困るの？」

HASにおいて以下を同時に満たす語彙設計が必要だった。

- 診断・評価・治療の文脈を立てないこと
- 行為の正解化（HowTo化）を防ぐこと
- 状態・位置・連鎖を観察可能に記述すること
- 実践現場で防衛・介入を誘発しないこと

## Forces
- 説明を簡単にすると「壊れる」が便利に使えてしまう
- しかし「壊れる」はエゴ的防衛・修復衝動を強く喚起する
- Failure Modes や比較表では、誤読耐性が特に求められる
- 行為や解決策を書くと、HASが即座にHowTo化する

## Decision
HAS全体において、以下の設計判断を採用する。

1. 「壊れる／壊さない／壊れやすい」を原則として使用しない  
2. 中心概念を **「選択の余地／選択可能性」** に統一する  
3. 状態は「結果」ではなく **連鎖・過程・位置** として記述する  
4. Failure Modes では **解決策・対応方法を一切提示しない**  
5. HASは「方法」ではなく **観察のための地図**として記述する  

## Rationale
- 「選択の余地」は当事者の体感と乖離しにくい
- 評価語ではなく、観察可能な事実として扱える
- 行為を正解化せず、判断保留・停止・撤退を自然に含む
- HASをHowToや成長モデルに誤読されるリスクを最小化できる

## Pros & Cons
- \+ 防衛・診断・介入の誤用を強く抑制できる
- \+ HASの非メソッド性・非操作性が明確になる
- \+ Architecture / Attunement Map の一貫性が高まる
- − 抽象度が上がり、即効性を求める読者は離脱しやすい
- − 説明に慣れが必要

## Note
- 「壊れる」は設計者視点では正確でも、実践文書では使用しない
- 「失敗」「成功」も原則として避ける
- 行為・対処・解決策は **書いた瞬間にHASをHowToに変質させる**
- HAS文書では「何が起きるか」までを書き、「何をすべきか」は書かない

## Alternatives Considered
- 「壊れる」をGlossaryで定義して使い続ける案  
  → 誤読耐性が低く却下
- 「失敗」に置き換える案  
  → 成長・学習モデルを呼び込むため却下
- 解決策付き Failure Modes  
  → メソッド化リスクが高く却下

## Consequences
- HASは「改善・成長・修復」から明確に距離を取る
- 読者は「どうするか」ではなく「今どこにいるか」を見る
- HASは再現性よりも **誤用耐性** を優先する体系として確定する
