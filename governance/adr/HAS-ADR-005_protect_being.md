# Title: HAS-ADR-005: HASを「Protocol」ではなく「System」として定義し、
# 構造分離によって思想の不変性と拡張性を両立させる

## Status
Accepted

## Context
HAS-ADR-004 により、HAS v2.0 は存在論・安全仕様として完成し、
思想的には「閉じた体系（Sealed）」として確定した。

一方で、以下の運用・設計上の課題が明確になった。

- 「Protocol（手順）」という表現が、HASを Doing（従う方法）として誤解させるリスク
- 実践例・場面集・テクニックの増加により、カーネル（思想）が汚染される懸念
- 権威化・布教化・規範化を、運用ではなく構造で防ぐ必要性
- LLM / 検索時代において、「概念」より「状況文」が入口になる現実

これらを踏まえ、HASを長期的に維持・公開するための
**名称・階層・責務分離の再定義**が必要となった。

## Problem
HASを社会に「置く」際に、

- 思想は不変でありながら
- 事例・応用・表現は増え続け
- しかも手順化・布教化・権威化しない

という要件を、どう構造的に保証するか？

## Decision
HASを **Protocol（手順）ではなく System（環境・制約系）** として明示的に定義する。

その上で、以下の三層構造を採用する。

### 1. Core（核）
- 存在論・憲法・禁忌を含む
- HAS v2.0 Final のみを配置
- **変更不可（Sealed）**

### 2. Governance（統治）
- HASを維持・保護するための内部ルール
- 権限は「創造・推奨」ではなく「拒否・保守」に限定
- 「Protocol」という語は、この層の内部規則にのみ使用する

### 3. Resources（接点）
- 場面集・パタン・クイックリファレンス等
- 実践・検索・発見のためのインターフェース
- 増殖・派生を許容（Fork可能）

また、Pattern を以下に分離する。

- `patterns/state/`  
  - 場の状態定義  
  - Doing を含まない
- `patterns/techniques/`  
  - 任意のテクニック  
  - HASの必須要件ではない

## Rationale
- 「System」という語は、Doingを要求せず、制約・環境・安全装置を自然に内包する
- Core を物理的に隔離することで、「完成した思想」を構造で保証できる
- Protocol を Governance 配下に閉じることで、外部への規範化・布教化を防げる
- State と Technique を分離することで、実践の拡張が思想汚染に直結しなくなる
- Resources を Interface として位置づけることで、
  「必要な人が、必要な時に、必要なだけ」発見できる導線が成立する

## Pros & Cons
**Pros**
+ **思想の固定:** カーネルが拡張・解釈競争に巻き込まれない
+ **誤用耐性:** 手順化・資格化・布教化を構造的に抑止できる
+ **拡張可能性:** 事例・場面・技法が増えても、HAS自体は変質しない
+ **検索適合性:** LLM / 検索エンジンに拾われやすい「状況ベースの入口」を設計できる

**Cons**
− **即効性の低下:** 「何をすればいいか」を即座に提示しない
− **理解コスト:** Systemとして理解するための抽象度が高い

## Alternatives Considered
- **HASをProtocolとして統一する案**
  - 理由: Doing中心の誤用・規範化・権威化を招くため不採用
- **CoreとResourcesを分離しない案**
  - 理由: 事例や技法の増加により、思想の境界が曖昧になるため不採用

## Consequences
HASは今後、

- 「従うべき方法」ではなく
- 「そこに在る安全装置（System）」

として提示・運用される。

HASの正当性は、
人物・肩書き・実践歴ではなく、
**Core（v2.0 Final）との整合性**によってのみ判断される。
