# ADR-008 READMEの役割限定とレイヤ構造の明確化

## Status
Accepted

## Context
HASの文書群が増えるにつれ、READMEが「仕様」「規範」「解釈の根拠」として読まれ、
README自体が権威化・誤用の起点になるリスクが顕在化した。

特に以下の問題が見えてきた：
- READMEの文言が、KernelやProtocolより優先されて解釈される
- README内の説明が事実上のルールとして運用される
- 「HASはDoingを定義しない」という原則が、運用レイヤまで誤適用される
- OS / TS の対立構造として誤読される

これらは、HASが忌避する「思想のメソッド化」「文書による統制」を引き起こす。

## Problem
READMEは何を担い、何を担わない文書とするべきか。  
また、HAS内部の文書・運用・実践の境界を、どこでどのように切るべきか。

## Forces
- READMEは最初に読まれるため、影響力が過剰になりやすい
- 詳細を書かないと「不親切」「不完全」と受け取られやすい
- 一方で、詳細を書くほど仕様化・ルール化が進む
- OS/TSの対比は説明力が高いが、善悪二元論に誤解されやすい

## Decision
以下を明示的に決定した：

1. READMEは**案内専用ドキュメント**とし、仕様・規範・判定根拠を持たせない  
2. README冒頭に「このREADMEは案内であり、仕様ではない」と明記する  
3. HAS全体を4レイヤ（core / docs / governance / resources）に分離し、
   READMEでは**構造と導線のみ**を示す  
4. 「HASはDoingを定義しない」という制約は、Kernel / Docs レイヤに限定する  
5. OS / TS は対立概念ではなく、目的関数の違いとして明示する  
6. 運用・判定・名称使用などのルールは、必ず Protocol / ADR に委譲する  

## Rationale
- READMEを仕様化すると、HASが最も避けたい「正解の提示」に転ぶ
- 境界を曖昧にすると、善意の実践者ほど誤用する
- レイヤ分離により、思想の純度と運用の現実を両立できる
- READMEを「地図」に徹することで、権威化を防げる

## Pros & Cons
- + 読者が「今どの文書を読むべきか」迷いにくくなる  
- + READMEが統制装置になるリスクを下げられる  
- + Kernel / Protocol の位置づけが明確になる  

- − READMEだけ読んでも全貌は理解できない  
- − 即効性・手順を求める読者には不親切に見える可能性がある  

## Note
- READMEに判断基準・例外条件・細則を書かないこと
- ルールが必要になったら、必ずProtocolまたはADRとして追加する
- README修正時は「案内以上の意味を持っていないか」を自己監査する

## Alternatives Considered
- READMEを包括仕様にする  
  - 却下理由：思想と運用が混線し、誤用時の歯止めが効かない
- READMEに簡易ルール集を載せる  
  - 却下理由：簡易であるほど独り歩きし、Protocolが参照されなくなる

## Consequences
- READMEは安定し、更新頻度が下がる
- 判断・運用の議論は、ADR / Protocol 側に集約される
- HASは「読めば使える体系」ではなく、「構造を理解して関わる体系」になる
